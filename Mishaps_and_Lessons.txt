Mishaps and Lessons Learned
============================

1. Fabricated npm Package Version
----------------------------------
Problem: The initial package.json specified @google-cloud/datastore@^9.3.0,
a version that does not exist. npm install failed on both VMs with
"ETARGET: No matching version found."

Root cause: The version number was made up instead of being verified against
the npm registry. The actual latest version was 10.1.0.

Fix: Ran `npm view @google-cloud/datastore version` to find the real version,
updated package.json, re-uploaded to both VMs.

Lesson: Always verify dependency versions against the actual registry before
writing them into a manifest. Never assume a version exists.


2. Ubuntu 22.04 Old Node.js + dpkg Conflicts
----------------------------------------------
Problem: Ubuntu 22.04's default nodejs package is v12, far too old for modern
npm packages. Attempting to install Node 18 via NodeSource caused dpkg
conflicts with pre-installed packages libnode-dev and libnode72, which owned
overlapping files like /usr/include/node/common.gypi and
/usr/share/systemtap/tapset/node.stp.

Root cause: The old system Node.js packages were not fully removed before
installing the NodeSource version, and dpkg refused to overwrite files owned
by other packages.

Fix: Removed libnode-dev first, then used `dpkg -i --force-overwrite` on the
cached NodeSource .deb to get past the libnode72 conflict.

Lesson: On Ubuntu with pre-installed Node.js, always purge old node packages
completely before installing from NodeSource. Or better yet, use nvm to
avoid system package conflicts entirely.


3. Insufficient VM Authentication Scopes
------------------------------------------
Problem: After deploying and starting the server, every Datastore call crashed
with "PERMISSION_DENIED: Request had insufficient authentication scopes."
The /greeting endpoint worked fine (no Datastore), but /register, /list,
and /clear all killed the server.

Root cause: GCP Compute Engine VMs are created with limited default OAuth
scopes that do not include Datastore access. The scope needed was
cloud-platform.

Fix: Stopped both VMs, ran `gcloud compute instances set-service-account
--scopes=cloud-platform`, then restarted them.

Lesson: When creating VMs that will access GCP services like Datastore,
always specify --scopes=cloud-platform at creation time. Fixing it later
requires stopping the VM, which also changes ephemeral external IPs.


4. IAM Role Missing After Scope Fix
--------------------------------------
Problem: After fixing scopes, the error changed to "PERMISSION_DENIED:
Missing or insufficient permissions." — a different error from a different
layer.

Root cause: The default Compute Engine service account did not have the
roles/datastore.user IAM role on the project. Scopes gate which APIs
the VM can call, but IAM roles gate whether the service account is
actually authorized.

Fix: Granted the datastore.user role to the default compute service account
with `gcloud projects add-iam-policy-binding`.

Lesson: GCP access requires BOTH correct OAuth scopes on the VM AND correct
IAM roles on the service account. They are independent checks and both must
pass.


5. Ephemeral IP Changed After VM Restart
------------------------------------------
Problem: Stopping instance-a to fix scopes caused its external IP to change
from 34.68.72.28 to 136.113.191.35. IPs.txt had to be updated.

Root cause: e2-micro instances use ephemeral IPs by default. Stopping and
starting a VM releases and reassigns a new ephemeral IP.

Lesson: If you need stable IPs, reserve static external IPs before creating
VMs. Otherwise, always re-check IPs after any stop/start cycle.


6. SSH Failures with Complex Command Strings
----------------------------------------------
Problem: gcloud compute ssh --command="..." worked for short commands like
"echo ok" but consistently returned exit code 255 for longer multi-statement
commands with semicolons, single quotes, and subshells.

Root cause: The layered quoting between the local shell, gcloud, and the
remote SSH session mangled the command string. Additionally, stale SSH
host keys after the IP change caused intermittent rejections.

Fix: Wrote the commands to a shell script (start.sh), uploaded it via
gcloud compute scp, then ran it with --command="bash ~/start.sh".

Lesson: For anything beyond trivial remote commands, upload a script file
and execute it. Fighting nested quoting across local shell -> gcloud -> SSH
is a losing battle.


7. nohup Background Process Killed by SSH Disconnect
------------------------------------------------------
Problem: Running `nohup node index.js &` inside a --command string appeared
to work (npm install output showed success) but the node process was not
running afterward. The server.log file did not exist.

Root cause: When the SSH session ends, background processes started in the
same session can be killed if they are not properly detached. The `disown`
builtin was needed, and the overall session lifetime was too short.

Fix: Combined nohup + disown in a dedicated script, added a sleep to let
the process fully detach before the SSH session closed.

Lesson: For long-running processes over SSH, use a startup script with
nohup + disown, or use tools like screen/tmux/systemd for reliable
process persistence.


Summary
========
Most of these issues boil down to three themes:
  (a) Never assume — verify versions, IPs, scopes, and permissions.
  (b) GCP access is two layers: scopes + IAM. Both must be configured.
  (c) Remote SSH command execution is fragile. Use script files.

Total time wasted on these issues: significant. But each one taught a
concrete lesson about cloud deployment that reading documentation alone
would not have made stick.
